## 이 실습에서 배운 것

<Steps>
  <Step title="io_uring의 SQ/CQ 흐름을 직접 구현했다">

1회차에서 liburing으로 SQE 제출과 CQE 수확을 직접 작성하여, 공유 메모리 기반 배치 I/O가 POSIX `read()` 대비 어떤 지연 개선을 가져오는지 체험했다. 코드를 한 줄씩 쓰면서 io_uring의 구조를 이해하는 것이 목적이었다.

  </Step>
  <Step title="fio로 체계적인 실험 매트릭스를 돌렸다">

2회차에서는 모드(interrupt, SQPOLL) x QD(1~256) x CPU 부하(idle, 50%, 100%) 조합을 fio로 측정했다. 셸 스크립트 한 장으로 수십 가지 조건을 재현 가능하게 돌리고, JSON 출력에서 지연/처리량/CPU 사용률을 추출하는 과학적 측정 방법을 익혔다.

  </Step>
  <Step title="Adaptive 정책을 C로 구현하고 fio baseline과 비교했다">

3회차에서는 fio가 제공하지 않는 adaptive 모드를 직접 C로 구현했다. CPU 사용률과 QD 두 신호만으로도 최적 전략 대비 85~97% 성능을 유지하면서 최악의 경우를 회피할 수 있었다. 커널 수정 없이 사용자 공간에서 구현 가능하다는 점에서 실용적 가치가 있다.

  </Step>
  <Step title="도구는 목적에 맞게 선택한다">

교육은 C, 측정은 fio, 연구는 C. 1회차에서 C로 io_uring을 직접 구현한 것은 구조 이해를 위해서였고, 2회차에서 fio로 전환한 것은 정밀한 실험 매트릭스를 효율적으로 돌리기 위해서였다. 3회차에서 다시 C로 돌아온 것은 fio가 지원하지 않는 정책을 구현해야 했기 때문이다. 모든 것을 하나의 도구로 하려 하면 비효율적이다.

  </Step>
</Steps>

## 한계와 향후 과제

이 연구에는 몇 가지 한계가 있다. 사용자 공간에서 `/proc/stat`을 읽는 방식은 100~200밀리초의 샘플링 지연이 불가피하여, 마이크로초 단위의 급격한 부하 변동에 대응하지 못한다. 3회차의 adaptive C와 fio 기준선은 측정 harness가 다르므로, 절대값 비교에는 한계가 있다. 모든 실험은 단일 NVMe 장치의 4KB 랜덤 읽기로 한정되었으며, 쓰기 워크로드나 혼합 I/O 패턴은 다루지 않았다.

향후 방향은 크게 세 갈래다. 첫째, 이 정책을 커널 모듈로 이식하면 인터럽트 핸들러 내부에서 모드를 전환할 수 있어 반응 지연이 수 마이크로초로 줄어든다. 둘째, IOPOLL을 결합하면 완료 경로의 인터럽트까지 제거하여 소프트웨어 경로의 오버헤드를 양방향에서 최소화할 수 있다. 셋째, 고정 임계값을 온라인 학습으로 대체하면 워크로드와 하드웨어 조합에 따라 정책이 스스로 최적화될 수 있다.

<Section title="Linux 셋업 — QEMU + KVM" id="setup-linux">

이 페이지는 [dpas-lab의 Linux 가이드](https://dpas-lab.pages.dev/setup-linux.html)를 기준으로 작성됐다.
DPAS 자체는 다루지 않으므로 "커널 5.18 업그레이드"와 "NVMe `poll_queues` 활성화" 단계는 제외했고, 그 외엔 원본 명령을 그대로 따라간다.
다른 OS는 [Setup 인덱스](/setup)로 돌아가서 선택한다.

## 0. 사전 확인

아키텍처와 KVM 가용성을 확인한다.

<Terminal title="아키텍처 및 KVM 확인" commands={[
  { cmd: "uname -m", output: ["x86_64"] },
  { cmd: "egrep -c '(vmx|svm)' /proc/cpuinfo", output: ["8"] },
  { cmd: "lsmod | grep kvm", output: ["kvm_intel  ...", "kvm  ..."] }
]} />

- `vmx`: Intel VT-x 지원
- `svm`: AMD-V 지원
- 결과가 0이거나 모듈이 안 보이면 KVM 비활성. 아래 주의사항 참조.

<Callout kind="warn">
**KVM 사용 불가 환경**
이미 VirtualBox/VMware VM 안에서 작업 중이면 nested KVM이 필요하다.
- VirtualBox: 머신 설정 → 시스템 → 가속 → "VT-x/AMD-V 중첩 페이징" 활성화
- VMware: VMX 파일에 `vhv.enable = "TRUE"` 추가 후 재시작

nested 활성화 확인:

```bash
cat /sys/module/kvm_intel/parameters/nested   # Y 이면 OK
cat /sys/module/kvm_amd/parameters/nested     # AMD의 경우
```

KVM을 못 쓰는 경우 `-enable-kvm`을 빼고 소프트웨어 에뮬레이션으로 실행 (느림).
</Callout>

### KVM 권한 설정

`qemu-system-x86_64` 실행 시 `permission denied /dev/kvm` 오류가 나면 현재 사용자를 kvm 그룹에 추가해야 한다.

<Terminal title="kvm 그룹 가입" commands={[
  { cmd: "sudo usermod -aG kvm $USER" },
  { cmd: "newgrp kvm" },
  { cmd: "ls -l /dev/kvm", output: ["crw-rw---- 1 root kvm  ...  /dev/kvm"] }
]} />

`group: kvm`으로 보이면 OK.

## 3. 디스크 이미지 생성

<Callout kind="note">
**디스크 분리 이유** — OS가 NVMe 위에 있으면 NVMe 드라이버 옵션을 바꿀 때 시스템이 멈출 수 있다.
OS는 virtio 디스크(`/dev/vda`)에, 실험 대상은 NVMe 디바이스(`/dev/nvme0n1`)에 분리해 둔다.
</Callout>

<Terminal title="디스크 이미지 생성" commands={[
  { cmd: "cd ~/qemu-lab" },
  { cmd: "qemu-img create -f qcow2 ubuntu.qcow2 30G" },
  { cmd: "qemu-img create -f raw nvme_disk.raw 16G" }
]} />

`ubuntu.qcow2`는 OS용 30GB qcow2(실제 크기는 사용량만큼 늘어남), `nvme_disk.raw`는 실험 대상 16GB raw.
raw 포맷은 `O_DIRECT`를 정확히 지원하기 위함이다.

## 4. Ubuntu 22.04 설치

<Callout kind="warn">
**WSL 사용자 주의** — WSL 환경에서는 `-nographic` 옵션이 설치 과정에서 문제를 일으킬 수 있다.
이 경우 `-nographic`을 제거하면 WSLg가 QEMU GUI 창을 자동으로 띄워 정상적으로 설치할 수 있다.
설치 완료 후 GRUB 설정에 `console=ttyS0`을 추가하면 이후 `-nographic` 모드를 쓸 수 있다:

```bash
sudo sed -i 's/^GRUB_CMDLINE_LINUX=""/GRUB_CMDLINE_LINUX="console=tty0 console=ttyS0,115200n8"/' /etc/default/grub
sudo update-grub
```
</Callout>

<CodeBlock language="bash" filename="install-vm-linux.sh">
{`cd ~/qemu-lab
qemu-system-x86_64 \\
  -m 4G -smp 4 \\
  -enable-kvm \\
  -drive file=ubuntu.qcow2,if=virtio,format=qcow2 \\
  -cdrom ubuntu-22.04-server-amd64.iso \\
  -boot d \\
  -netdev user,id=net0,hostfwd=tcp::2222-:22 -device virtio-net-pci,netdev=net0 \\
  -nographic`}
</CodeBlock>

<Callout kind="warn">
**Ubuntu 서버 설치 시 주의사항**

- 언어·키보드·네트워크는 기본값
- **`openssh-server` 반드시 선택** (스페이스바로 체크)
- Storage: LVM 설정 시 디스크 전체를 사용하도록 설정. 기본값이 절반만 할당하므로, 설치 후 다음 명령으로 확장하거나 설치 중 "Custom storage layout"에서 전체 크기를 할당:

  ```bash
  sudo lvextend -l +100%FREE /dev/ubuntu-vg/ubuntu-lv
  sudo resize2fs /dev/ubuntu-vg/ubuntu-lv
  ```
- 추가 패키지: 선택 안 해도 됨
- 설치 완료 후 "Reboot Now" → 콘솔에서 `Ctrl-A` + `X`로 QEMU 종료
</Callout>

## 5. 설치 후 VM 기동 (NVMe 포함)

설치 완료 후 `-cdrom`을 빼고 NVMe 디스크를 붙여서 기동한다.

<CodeBlock language="bash" filename="run-vm-linux.sh">
{`cd ~/qemu-lab
qemu-system-x86_64 \\
  -m 4G -smp 4 \\
  -enable-kvm \\
  -drive file=ubuntu.qcow2,if=virtio,format=qcow2 \\
  -drive file=nvme_disk.raw,id=nvm,if=none,format=raw \\
  -device nvme,id=nvme0,serial=deadbeef \\
  -device nvme-ns,drive=nvm,bus=nvme0,nsid=1,logical_block_size=512,physical_block_size=512 \\
  -netdev user,id=net0,hostfwd=tcp::2222-:22 -device virtio-net-pci,netdev=net0 \\
  -nographic`}
</CodeBlock>

VM 부팅 완료 후 SSH 접속:

<Terminal title="SSH 접속" commands={[
  { cmd: "ssh -p 2222 <username>@localhost" }
]} />

처음 접속 시 host key 경고가 나오면 `yes` 입력.

<Callout kind="note">
이전에 다른 VM이 같은 포트(2222)에 있었다면 호스트 키 충돌로 SSH가 거부될 수 있다.
그때는 다음 명령으로 옛 키를 지우고 다시 접속한다.

```bash
ssh-keygen -R "[localhost]:2222"
```
</Callout>

### 옵션 한 줄 해설

위 두 명령(설치 단계 4와 기동 단계 5)에 등장한 모든 QEMU 옵션을 정리한다.
같은 옵션이 두 단계에 공통이고, 차이는 ISO 관련 옵션(`-cdrom`, `-boot d`)과 NVMe 디바이스 옵션뿐이다.

**리소스 할당 (공통)**

- `-m 4G` — 게스트에 할당할 RAM 크기. 4GB는 Ubuntu Server + apt 패키지 + io_uring 빌드에 충분.
- `-smp 4` — 게스트에 노출할 vCPU 수. 호스트 코어가 적으면 2로 줄여도 된다.
- `-enable-kvm` — Linux의 KVM으로 하드웨어 가속. `/dev/kvm`에 접근 권한이 있어야 한다 (kvm 그룹 가입 필요).

**OS 디스크 (공통)**

- `-drive file=ubuntu.qcow2,if=virtio,format=qcow2` — Ubuntu가 설치되는 OS 디스크.
  - `if=virtio` — virtio 인터페이스로 게스트에 노출 (`/dev/vda`로 보임). NVMe와 분리하는 핵심.
  - `format=qcow2` — Copy-on-Write 포맷. 30GB로 만들어도 실제 사용량만큼만 디스크를 점유.

**ISO 관련 (4단계 설치 시에만)**

- `-cdrom <iso>` — ISO 파일을 가상 CD-ROM으로 마운트.
- `-boot d` — CD-ROM(`d`)에서 부팅하라는 명시적 지정. 빠지면 빈 OS 디스크에서 부팅을 시도해 실패한다.

**NVMe 디스크 (5단계 기동 시에만)**

NVMe는 세 줄로 나눠서 정의한다. 각 줄이 디스크 스택의 다른 레이어를 담당한다.

- `-drive file=nvme_disk.raw,id=nvm,if=none,format=raw` — 백킹 스토리지 등록.
  - `if=none` — 어떤 인터페이스에도 자동 연결하지 말고 ID만 등록. 다음 줄의 `-device`가 이걸 가져다 쓴다.
  - `format=raw` — 그대로의 raw 디스크. `O_DIRECT`가 정확히 동작하려면 raw가 필수.
- `-device nvme,id=nvme0,serial=deadbeef` — NVMe 컨트롤러 디바이스 추가.
  - `serial=deadbeef` — 임의 식별자 (필수 필드, 값 자체는 의미 없음).
- `-device nvme-ns,drive=nvm,bus=nvme0,nsid=1,logical_block_size=512,physical_block_size=512` — 컨트롤러 아래에 namespace를 만들고 위 `nvm` 드라이브를 연결.
  - `nsid=1` — namespace ID. 게스트에서 `/dev/nvme0n1`의 마지막 `1`이 이 값.
  - `logical_block_size`, `physical_block_size` — 섹터 크기. 512는 가장 호환성 좋은 값.

이 세 줄이 모두 있어야 게스트에 `/dev/nvme0n1`이 보인다.
하나라도 빠지면 디바이스 인식이 안 된다.

**네트워크 (공통)**

- `-netdev user,id=net0,hostfwd=tcp::2222-:22` — user-mode 네트워크 백엔드(NAT). 게스트가 호스트 인터넷을 빌려 쓴다. `id=net0`는 다음 device가 가져다 쓸 식별자.
- `-device virtio-net-pci,netdev=net0` — 게스트에 virtio-net 가상 NIC를 만들고 위 백엔드와 연결. virtio 드라이버는 Ubuntu 커널에 기본 포함이라 게스트가 인터페이스를 즉시 인식한다.
- `hostfwd=tcp::2222-:22` — 호스트 2222 포트를 게스트의 22(SSH)로 포워딩. 이 덕분에 호스트에서 `ssh -p 2222 user@localhost`로 게스트에 접속 가능.

레거시 `-net nic -net user,...` 문법도 동작은 하지만, 게스트에 e1000 NIC을 만들어 부팅 시 `Wait for Network to be Configured`로 30~120초 멈추는 케이스가 잦다. 위 `-netdev + virtio-net-pci` 형태가 안정적이다.

**디스플레이 (공통)**

- `-nographic` — 그래픽 출력 없이 시리얼 콘솔을 현재 터미널에 직접 연결.
  - 별도 GUI 창이 뜨지 않아 원격 Linux 서버처럼 다룰 수 있다.
  - 종료는 `Ctrl-A` + `X` (시리얼 콘솔에서의 QEMU 단축키).
  - WSL 환경에서만 설치 단계에 문제가 될 수 있어 위 경고 박스 참조.

## 편의 스크립트

반복 기동을 위해 부팅 명령을 스크립트로 저장해 두면 편하다.

<CodeBlock language="bash" filename="~/qemu-lab/start-vm.sh">
{`#!/bin/bash
cd ~/qemu-lab
qemu-system-x86_64 \\
  -m 4G -smp 4 \\
  -enable-kvm \\
  -drive file=ubuntu.qcow2,if=virtio,format=qcow2 \\
  -drive file=nvme_disk.raw,id=nvm,if=none,format=raw \\
  -device nvme,id=nvme0,serial=deadbeef \\
  -device nvme-ns,drive=nvm,bus=nvme0,nsid=1,logical_block_size=512,physical_block_size=512 \\
  -netdev user,id=net0,hostfwd=tcp::2222-:22 -device virtio-net-pci,netdev=net0 \\
  -nographic "$@"`}
</CodeBlock>

<Terminal title="실행 권한 부여 + alias 등록" commands={[

## 트러블슈팅

| 증상 | 원인 | 해결 |
|---|---|---|
| `Could not access KVM kernel module: No such file or directory` | KVM 모듈 미로드 또는 권한 없음 | `sudo modprobe kvm_intel` (또는 `kvm_amd`), `sudo usermod -aG kvm $USER` |
| `permission denied /dev/kvm` | kvm 그룹 미등록 | `newgrp kvm` 후 재시도 |
| `nvme0n1` 디바이스 없음 | `-device nvme` 옵션 누락 | 5단계 명령의 NVMe 세 줄 확인 |
| ISO 부팅 실패 (UEFI 셸로 떨어짐) | `-boot d` 누락 | 4단계 명령에 `-boot d` 포함 확인 |
| SSH 접속 거부 | `openssh-server` 미설치 | 게스트에서 `sudo apt install openssh-server` 후 VM 재시작 |
| SSH `WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED` | 이전 VM의 호스트 키와 충돌 | `ssh-keygen -R "[localhost]:2222"` |
| 부팅 시 `Wait for Network to be Configured` 30~120초 정지 / 게스트에서 `apt install`이 DNS 오류로 실패 | 레거시 `-net nic`로 e1000 NIC을 받았는데 netplan이 인터페이스를 못 잡음 | (1) QEMU 명령을 위 `-netdev user,id=net0,... -device virtio-net-pci,netdev=net0` 형태로 교체. (2) 이미 설치된 VM이라면 게스트에서 `ip a`로 인터페이스 이름 확인 → `/etc/netplan/00-installer-config.yaml`에서 인터페이스 이름을 실제 이름으로 수정 → `sudo netplan apply`. (3) 부팅 대기를 단축하려면 `sudo systemctl disable systemd-networkd-wait-online.service` |
| nested KVM 미지원 | VM 안의 VM 환경 | 호스트 hypervisor에서 nested 가상화 활성화, 또는 `-enable-kvm` 제거 |

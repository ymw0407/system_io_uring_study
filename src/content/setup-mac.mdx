<Section title="macOS 셋업 — QEMU + HVF" id="setup-mac">

이 페이지는 [dpas-lab의 macOS 가이드](https://dpas-lab.pages.dev/setup-mac.html)를 기준으로 작성됐다.
DPAS 자체는 다루지 않으므로 "커널 5.18 업그레이드"와 "NVMe `poll_queues` 활성화" 단계는 제외했고, 그 외엔 원본 명령을 그대로 따라간다.
다른 OS는 [Setup 인덱스](/setup)로 돌아가서 선택한다.

## 0. 사전 확인 — Mac 아키텍처

<Terminal title="아키텍처 확인" commands={[
  { cmd: "uname -m", output: ["arm64    # Apple Silicon (M1/M2/M3/M4)", "x86_64   # Intel Mac"] }
]} />

<Callout kind="warn">
**Apple Silicon 주의** — ARM 아키텍처라서 `qemu-system-x86_64`를 HVF로 가속할 수 없다.
`-accel hvf: invalid accelerator hvf` 같은 오류가 나오면 이 케이스다.
Apple Silicon에서는 ARM64 Ubuntu + `qemu-system-aarch64` 조합으로 진행한다.
</Callout>

## 3. 디스크 이미지 생성

<Callout kind="note">
**디스크 분리 이유** — OS가 NVMe 위에 있으면 NVMe 드라이버 옵션을 바꿀 때 시스템이 멈출 수 있다.
OS는 virtio 디스크(`/dev/vda`)에, 실험 대상은 NVMe 디바이스(`/dev/nvme0n1`)에 분리해 둔다.
</Callout>

<Terminal title="디스크 이미지 생성" commands={[
  { cmd: "cd ~/qemu-lab" },
  { cmd: "qemu-img create -f qcow2 ubuntu.qcow2 30G" },
  { cmd: "qemu-img create -f raw nvme_disk.raw 16G" }
]} />

`ubuntu.qcow2`는 OS용 30GB qcow2(실제 크기는 사용량만큼만 늘어남), `nvme_disk.raw`는 실험 대상 16GB raw.
raw 포맷은 `O_DIRECT`를 정확히 지원하기 위함이다.

## 4. Ubuntu 설치

이 단계에서는 ISO에서 부팅해 OS 디스크에 설치만 한다.
NVMe 디스크는 아직 붙이지 않는다.

### Intel Mac

<CodeBlock language="bash" filename="install-vm-intel.sh">
{`cd ~/qemu-lab
qemu-system-x86_64 \\
  -m 4G -smp 4 \\
  -accel hvf \\
  -drive file=ubuntu.qcow2,if=virtio,format=qcow2 \\
  -cdrom ubuntu-22.04-server-amd64.iso \\
  -boot d \\
  -netdev user,id=net0,hostfwd=tcp::2222-:22 -device virtio-net-pci,netdev=net0 \\
  -nographic`}
</CodeBlock>

### Apple Silicon

<Terminal title="EDK2 펌웨어 경로 확인" commands={[
  { cmd: "ls $(brew --prefix)/share/qemu/edk2-aarch64-code.fd" }
]} />

<CodeBlock language="bash" filename="install-vm-silicon.sh">
{`cd ~/qemu-lab
qemu-system-aarch64 \\
  -M virt \\
  -cpu host \\
  -m 4G -smp 4 \\
  -accel hvf \\
  -bios $(brew --prefix)/share/qemu/edk2-aarch64-code.fd \\
  -drive file=ubuntu.qcow2,if=virtio,format=qcow2 \\
  -cdrom ubuntu-22.04-server-arm64.iso \\
  -boot d \\
  -netdev user,id=net0,hostfwd=tcp::2222-:22 -device virtio-net-pci,netdev=net0 \\
  -nographic`}
</CodeBlock>

`-nographic`이라 별도 GUI 창이 뜨지 않고 현재 터미널에 시리얼 콘솔이 그대로 출력된다.
`-boot d`가 ISO에서 부팅하라는 명시적 지정이다.

<Callout kind="warn">
**Ubuntu 서버 설치 시 주의사항**

- 언어·키보드·네트워크는 기본값
- **`openssh-server` 반드시 선택** (이후 SSH 접속에 필요)
- Storage: LVM 설정 시 디스크 전체를 사용하도록 설정.
  기본값이 절반만 할당하므로, 설치 후 다음 명령으로 확장하거나 설치 중 "Custom storage layout"에서 전체 크기를 할당:

  ```bash
  sudo lvextend -l +100%FREE /dev/ubuntu-vg/ubuntu-lv
  sudo resize2fs /dev/ubuntu-vg/ubuntu-lv
  ```
- 재부팅 후 ISO 제거 (다음 단계의 명령에 `-cdrom`이 빠져 있다)
</Callout>

VM이 부팅을 마치고 `login:` 프롬프트가 뜨면, 새 터미널에서 SSH로 접속한다.

<Terminal title="SSH 접속" commands={[
  { cmd: "ssh -p 2222 <username>@localhost" }
]} />

<Callout kind="note">

```bash

```
</Callout>

### 옵션 한 줄 해설

위 두 명령(설치 단계 4와 기동 단계 5)에 등장한 모든 QEMU 옵션을 정리한다.
같은 옵션이 두 단계에 공통이고, 차이는 ISO 관련 옵션(`-cdrom`, `-boot d`)과 NVMe 디바이스 옵션뿐이다.

**머신 종류와 CPU (Apple Silicon에서만 명시 필요)**

- `-M virt` — ARM64 가상 머신 타입. ARM은 PC가 아니므로 가상 SoC 사양(virt 머신)을 명시적으로 지정해야 한다. x86_64는 자동으로 PC 머신이 선택돼 생략 가능.
- `-cpu host` — 호스트 CPU의 기능을 그대로 게스트에 노출. HVF 가속이 동작하려면 호스트와 같은 CPU 모델을 게스트에 보여야 한다.
- `-bios $(brew --prefix)/share/qemu/edk2-aarch64-code.fd` — UEFI 펌웨어 파일. ARM64 virt 머신은 BIOS가 자동 선택되지 않아 EDK2 UEFI를 직접 지정해야 부팅 매니저가 동작한다. x86_64는 SeaBIOS가 자동.

**리소스 할당 (공통)**

- `-m 4G` — 게스트에 할당할 RAM 크기. 4GB는 Ubuntu Server + apt 패키지 + io_uring 빌드에 충분.
- `-smp 4` — 게스트에 노출할 vCPU 수. 호스트 코어가 적으면 2로 줄여도 된다.
- `-accel hvf` — Apple Hypervisor.framework로 하드웨어 가속. **호스트와 같은 아키텍처일 때만 동작**(Apple Silicon은 ARM 게스트, Intel Mac은 x86_64 게스트).

**OS 디스크 (공통)**

- `-drive file=ubuntu.qcow2,if=virtio,format=qcow2` — Ubuntu가 설치되는 OS 디스크.
  - `if=virtio` — virtio 인터페이스로 게스트에 노출 (게스트 안에서 `/dev/vda`로 보임). NVMe와 분리하는 핵심.
  - `format=qcow2` — Copy-on-Write 포맷. 30GB로 만들어도 실제 사용량만큼만 디스크를 점유한다.

**ISO 관련 (4단계 설치 시에만)**

- `-cdrom <iso>` — ISO 파일을 가상 CD-ROM으로 마운트.
- `-boot d` — CD-ROM(`d`)에서 부팅하라는 명시적 지정. 이게 빠지면 QEMU가 빈 `ubuntu.qcow2`에서 부팅하려다 UEFI 셸로 떨어진다.

**NVMe 디스크 (5단계 기동 시에만)**

NVMe는 세 줄로 나눠서 정의한다. 각 줄이 디스크 스택의 다른 레이어를 담당한다.

- `-drive file=nvme_disk.raw,id=nvm,if=none,format=raw` — 백킹 스토리지 등록.
  - `if=none` — 어떤 인터페이스에도 자동 연결하지 말고 ID만 등록. 다음 줄의 `-device`가 이걸 가져다 쓴다.
  - `format=raw` — 그대로의 raw 디스크. `O_DIRECT`가 정확히 동작하려면 raw가 필수.
- `-device nvme,id=nvme0,serial=deadbeef` — NVMe 컨트롤러 디바이스 추가.
  - `serial=deadbeef` — 임의 식별자(필수 필드, 값 자체는 의미 없음).
- `-device nvme-ns,drive=nvm,bus=nvme0,nsid=1,logical_block_size=512,physical_block_size=512` — 컨트롤러 아래에 namespace를 만들고 위 `nvm` 드라이브를 연결.
  - `nsid=1` — namespace ID. 게스트에서 `/dev/nvme0n1`의 마지막 `1`이 이 값.
  - `logical_block_size`, `physical_block_size` — 섹터 크기. 512는 가장 호환성 좋은 값.

이 세 줄이 모두 있어야 게스트에 `/dev/nvme0n1`이 보인다.
하나라도 빠지면 디바이스 인식이 안 된다.

**네트워크 (공통)**

- `-netdev user,id=net0,hostfwd=tcp::2222-:22` — user-mode 네트워크 백엔드(NAT). 게스트가 호스트 인터넷을 빌려 쓴다. `id=net0`는 다음 device가 가져다 쓸 식별자.
- `-device virtio-net-pci,netdev=net0` — 게스트에 virtio-net 가상 NIC를 만들고 위 백엔드와 연결. virtio 드라이버는 Ubuntu 커널에 기본 포함이라 게스트가 인터페이스를 즉시 인식한다.
- `hostfwd=tcp::2222-:22` — 호스트의 2222 포트를 게스트의 22(SSH)로 포워딩. 이 덕분에 호스트에서 `ssh -p 2222 user@localhost`로 게스트에 접속 가능.

레거시 `-net nic -net user,...` 문법도 동작은 하지만, 게스트에 e1000 NIC을 만들어 부팅 시 `Wait for Network to be Configured`로 30~120초 멈추는 케이스가 잦다. 위 `-netdev + virtio-net-pci` 형태가 안정적이다.

**디스플레이 (공통)**

- `-nographic` — 그래픽 출력 없이 시리얼 콘솔을 현재 터미널에 직접 연결.
  - 별도의 Cocoa 창이 뜨지 않아 원격 Linux 서버처럼 다룰 수 있다.
  - 종료는 `Ctrl-A` + `X` (시리얼 콘솔에서의 QEMU 단축키).
  - 이걸 빼고 그래픽 디바이스도 명시 안 하면 QEMU monitor 창만 떠서 화면이 안 나오는 것처럼 보인다.

## 트러블슈팅

| 증상 | 원인 | 해결 |
|---|---|---|
| `Could not access KVM kernel module` | macOS에서 `-enable-kvm` 사용 | `-accel hvf`로 변경 |
| `hvf: Error initializing HVF` 또는 `invalid accelerator hvf` | Apple Silicon에서 x86_64 VM 시도 | ARM64 ISO와 `qemu-system-aarch64`로 전환 |
| QEMU monitor만 뜨고 디스플레이 안 나옴 | `-nographic` 누락 또는 그래픽 디바이스 없음 | `-nographic` 사용 또는 `-device virtio-gpu-pci` 추가 |
| ISO 부팅 실패 (UEFI 셸로 떨어짐) | `-boot d` 누락 | 4단계 명령에 `-boot d` 포함 확인 |
| `nvme0n1` 디바이스 없음 | NVMe `-device` 옵션 누락 | 5단계의 `nvme,nvme-ns` 옵션 확인 |
| SSH `WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED` | 이전 VM의 호스트 키와 충돌 | `ssh-keygen -R "[localhost]:2222"` |
| SSH 접속 불가 | VM 부팅 중 또는 openssh-server 미설치 | 30초 대기 후 재시도, 또는 설치 위저드 다시 확인 |
| 부팅 시 `Wait for Network to be Configured` 30~120초 정지 / 게스트에서 `apt install`이 DNS 오류로 실패 | 레거시 `-net nic`로 e1000 NIC을 받았는데 netplan이 인터페이스를 못 잡음 | (1) QEMU 명령을 위 `-netdev user,id=net0,... -device virtio-net-pci,netdev=net0` 형태로 교체. (2) 이미 설치된 VM이라면 게스트에서 `ip a`로 인터페이스 이름 확인 → `/etc/netplan/00-installer-config.yaml`에서 인터페이스 이름을 실제 이름으로 수정 → `sudo netplan apply`. (3) 부팅 대기를 단축하려면 `sudo systemctl disable systemd-networkd-wait-online.service` |
| `edk2-aarch64-code.fd` 없음 | QEMU brew 버전 문제 | `brew reinstall qemu` |

**준비 완료 체크리스트 (macOS)**

- 호스트에 `brew install qemu` 완료
- `uname -m`으로 자기 Mac 확인 (arm64 또는 x86_64)
- 칩에 맞는 Ubuntu 22.04 Server ISO 다운로드 완료
- `qemu-img`로 OS 디스크와 NVMe 디스크 생성
- ISO로 부팅해 Ubuntu 설치 완료 (openssh-server 포함, LVM 전체 디스크 할당)
- ISO 없이 NVMe 포함 부팅 후 SSH 접속 가능
- 게스트에서 `sudo apt install liburing-dev` 완료
- `uname -r`이 5.1 이상
- `pkg-config --modversion liburing`이 2.x
- `lsblk`에 `vda`와 `nvme0n1` 모두 보임
- `check` 프로그램이 "io_uring 준비 완료: SQ entries = 8" 출력

위 항목이 모두 통과하면 Part 2로 진행할 수 있다.

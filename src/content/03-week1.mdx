## 이번 실습의 목표

이번 실습의 목표는 벤치마크 수치가 아니다.
io_uring의 SQ/CQ 흐름을 직접 C로 짜서, "내가 io_uring으로 파일을 읽었다"는 경험을 얻는 것이다.
`clock_gettime`으로 대강의 latency만 확인하면 충분하다.

## io_uring 흐름 직접 구현하기

앞 절에서 배운 io_uring의 다섯 단계를 실제 C 코드로 옮긴다.
각 단계가 어떤 의미인지 하나씩 짚으며 읽어 보자.

1. `io_uring_get_sqe()` -- SQ에서 빈 슬롯을 하나 받는다. 요청서 양식을 하나 꺼내는 것이다.
2. `io_uring_prep_read()` -- "이 파일의 이 위치에서 4KB를 읽어라"라고 양식을 채운다.
3. `io_uring_submit()` -- 채워 둔 SQE를 커널에 제출한다. 시스템 콜은 이 한 번뿐이다.
4. `io_uring_wait_cqe()` -- CQ에 결과가 도착할 때까지 기다린다. 커널이 읽기를 완료하면 CQE를 남긴다.
5. `io_uring_cqe_seen()` -- 결과를 확인했음을 커널에 알린다. 이래야 해당 CQ 슬롯이 재사용된다.

아래 코드는 이 다섯 단계를 100번 반복하여 4KB 랜덤 읽기의 평균 지연을 출력한다.
QD 튜닝이나 배칭 최적화 없이, 한 번에 하나씩 요청하는 가장 단순한 구조다.

<CodeBlock language="c" filename="uring_read.c">
{`#include <liburing.h>
#include <fcntl.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <time.h>
#include <unistd.h>

#define BS    4096
#define COUNT 100

static double timespec_diff_us(struct timespec *a,
                               struct timespec *b)
{
    return (b->tv_sec - a->tv_sec) * 1e6
         + (b->tv_nsec - a->tv_nsec) / 1e3;
}

int main(int argc, char *argv[])
{
    if (argc < 2) {
        fprintf(stderr,
                "usage: %s <file>\\n", argv[0]);
        return 1;
    }

    int fd = open(argv[1], O_RDONLY | O_DIRECT);
    if (fd < 0) {
        perror("open");
        return 1;
    }

    struct io_uring ring;
    if (io_uring_queue_init(1, &ring, 0) < 0) {
        perror("io_uring_queue_init");
        close(fd);
        return 1;
    }

    /* O_DIRECT requires aligned buffer */
    void *buf;
    if (posix_memalign(&buf, BS, BS) != 0) {
        perror("posix_memalign");
        return 1;
    }

    struct timespec t0, t1;
    clock_gettime(CLOCK_MONOTONIC, &t0);

    for (int i = 0; i < COUNT; i++) {
        off_t off = (rand() % 8192) * (off_t)BS;

        /* step 1: get empty SQE slot */
        struct io_uring_sqe *sqe;
        sqe = io_uring_get_sqe(&ring);

        /* step 2: fill in the read request */
        io_uring_prep_read(sqe, fd, buf, BS, off);

        /* step 3: submit to kernel */
        io_uring_submit(&ring);

        /* step 4: wait for completion */
        struct io_uring_cqe *cqe;
        if (io_uring_wait_cqe(&ring, &cqe) < 0) {
            perror("io_uring_wait_cqe");
            break;
        }
        if (cqe->res < 0) {
            fprintf(stderr, "read error: %s\\n",
                    strerror(-cqe->res));
            break;
        }

        /* step 5: mark CQE as consumed */
        io_uring_cqe_seen(&ring, cqe);
    }

    clock_gettime(CLOCK_MONOTONIC, &t1);
    double total_us = timespec_diff_us(&t0, &t1);
    printf("count=%d, total=%.1f us, "
           "avg=%.1f us/op\\n",
           COUNT, total_us, total_us / COUNT);

    free(buf);
    io_uring_queue_exit(&ring);
    close(fd);
    return 0;
}`}
</CodeBlock>

## POSIX read()와 비교해 보기

같은 작업을 전통적인 `read()` 시스템 콜로도 짜 두면 비교 기준이 생긴다.
아래는 동일한 조건(4KB 랜덤 읽기 100회, `O_DIRECT`)으로 `read()`를 호출하는 코드다.

<CodeBlock language="c" filename="posix_read.c">
{`#define _GNU_SOURCE
#include <fcntl.h>
#include <stdio.h>
#include <stdlib.h>
#include <time.h>
#include <unistd.h>

#define BS    4096
#define COUNT 100

static double timespec_diff_us(struct timespec *a,
                               struct timespec *b)
{
    return (b->tv_sec - a->tv_sec) * 1e6
         + (b->tv_nsec - a->tv_nsec) / 1e3;
}

int main(int argc, char *argv[])
{
    if (argc < 2) {
        fprintf(stderr,
                "usage: %s <file>\\n", argv[0]);
        return 1;
    }

    int fd = open(argv[1], O_RDONLY | O_DIRECT);
    if (fd < 0) {
        perror("open");
        return 1;
    }

    void *buf;
    if (posix_memalign(&buf, BS, BS) != 0) {
        perror("posix_memalign");
        return 1;
    }

    struct timespec t0, t1;
    clock_gettime(CLOCK_MONOTONIC, &t0);

    for (int i = 0; i < COUNT; i++) {
        off_t off = (rand() % 8192) * (off_t)BS;
        if (pread(fd, buf, BS, off) != BS) {
            perror("pread");
            return 1;
        }
    }

    clock_gettime(CLOCK_MONOTONIC, &t1);
    double total_us = timespec_diff_us(&t0, &t1);
    printf("count=%d, total=%.1f us, "
           "avg=%.1f us/op\\n",
           COUNT, total_us, total_us / COUNT);

    free(buf);
    close(fd);
    return 0;
}`}
</CodeBlock>

  { cmd: "./posix_read /tmp/testfile", output: ["count=100, total=12255.8 us, avg=122.6 us/op"] }
]} />

QD=1 환경에서는 두 방식의 평균 지연이 거의 구분되지 않는다.
한 번에 하나씩 요청해 매번 완료까지 기다리는 패턴이라 io_uring의 핵심 이점인 *배칭*과 *비동기 제출*이 작동할 여지가 없기 때문이다.
이 구간에서는 SQ/CQ를 거치는 비용이 `read()` 한 번의 비용과 사실상 같아진다.

<Callout kind="tip">
여기서 보여주려는 건 우열이 아니라 io_uring이 같은 일을 다른 경로로 처리한다는 사실이다.
QD를 높여 여러 요청을 한 번에 깔아 두는 시점부터 두 방식의 곡선이 갈라지는데, 이것은 다음 실습에서 fio로 직접 측정한다.
</Callout>

## 여기까지 이해한 것

이번 실습에서 직접 확인한 것은 세 가지다.
첫째, io_uring의 SQ/CQ 흐름을 C 코드로 구현하고 실제로 파일을 읽었다.
둘째, `get_sqe`, `prep_read`, `submit`, `wait_cqe`, `cqe_seen` 다섯 단계가 io_uring의 기본 사이클이라는 것을 코드로 확인했다.
셋째, QD=1 단발 호출 환경에서는 POSIX `read()`와 io_uring의 평균 지연이 거의 같다는 것을 직접 측정했다. io_uring의 이점은 이 구간에서 드러나지 않는다.

아직 QD를 1로 고정했기 때문에 io_uring의 진짜 강점인 배칭과 비동기 처리는 활용하지 못했다.
다음 회차에서는 fio를 사용하여 QD, SQPOLL, CPU 부하를 변수로 두고 정밀하게 측정한다.

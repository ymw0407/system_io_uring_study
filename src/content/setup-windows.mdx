<Section title="Windows 셋업 — WSL2 또는 WHPX" id="setup-windows">

이 페이지는 [dpas-lab의 Windows 가이드](https://dpas-lab.pages.dev/setup-windows.html)를 기준으로 작성됐다.
DPAS 자체는 다루지 않으므로 "커널 5.18 업그레이드"와 "NVMe `poll_queues` 활성화" 단계는 제외했고, 그 외엔 원본 명령을 그대로 따라간다.
다른 OS는 [Setup 인덱스](/setup)로 돌아가서 선택한다.

## 0. 사전 확인

### 시스템 요구사항

- Windows 10 (버전 1903 이상) 또는 Windows 11
- RAM 8GB 이상 권장 (VM에 4GB 할당)
- 여유 디스크 60GB 이상

### 가상화 지원 확인

작업 관리자 → 성능 → CPU → 가상화: **사용** 인지 확인.
또는 PowerShell에서:

<Terminal title="가상화 지원 확인 (PowerShell)" commands={[
  { cmd: "Get-ComputerInfo -Property HyperVisorPresent, HyperVRequirementVirtualizationFirmwareEnabled" }
]} />

가상화가 비활성화된 경우 BIOS/UEFI에서 **Intel VT-x** 또는 **AMD-V**를 활성화해야 한다.

## 두 가지 방법 선택

| 방법 | 특징 | 권장 대상 |
|---|---|---|
| **방법 1: WSL2 + QEMU** | Linux 명령어 환경, KVM 가속 지원 | 개발 경험 있는 학생 (권장) |
| **방법 2: Windows 네이티브 QEMU** | Windows에서 직접 실행, WHPX 가속 | WSL2 없는 환경 |

## 방법 1: WSL2 + QEMU (권장)

WSL2 내부에서 QEMU를 실행하면 KVM 가속을 사용할 수 있어 성능이 좋다.

### 1-1. WSL2 설치

PowerShell을 **관리자 권한**으로 실행:

<Terminal title="WSL2 설치" commands={[
  { cmd: "wsl --install" }
]} />

재부팅 후 Ubuntu 22.04 설정 완료 (사용자명·비밀번호 입력).

이미 WSL1이 설치된 경우 WSL2로 업그레이드:

<Terminal title="WSL1 → WSL2 업그레이드" commands={[
  { cmd: "wsl --set-default-version 2" },
  { cmd: "wsl --set-version Ubuntu 2" }
]} />

버전 확인:

<Terminal title="WSL 버전 확인" commands={[
  { cmd: "wsl --list --verbose", output: ["NAME      STATE   VERSION", "Ubuntu    Running 2          # 2 이어야 함"] }
]} />

### 1-2. WSL2 내부에서 KVM 확인

WSL2 터미널(Ubuntu)을 열고:

<Terminal title="KVM 장치 확인" commands={[
  { cmd: "ls /dev/kvm" }
]} />

`/dev/kvm`이 없는 경우, PowerShell 관리자 권한으로 다음을 실행:

<Terminal title="가상화 기능 활성화 (PowerShell 관리자)" commands={[
  { cmd: "dism.exe /online /enable-feature /featurename:VirtualMachinePlatform /all /norestart" },
  { cmd: "dism.exe /online /enable-feature /featurename:Microsoft-Hyper-V /all /norestart" }
]} />

재부팅 후 WSL2 재시작:

<Terminal title="WSL2 재시작" commands={[
  { cmd: "wsl --shutdown" },
  { cmd: "wsl" }
]} />

### 1-3. WSL2에서 QEMU 설치

<Terminal title="QEMU 설치 (WSL2 안에서)" commands={[
  { cmd: "sudo apt update" },
  { cmd: "sudo apt install -y qemu-system-x86 qemu-utils" },
  { cmd: "qemu-system-x86_64 --version" }
]} />

### 1-4. KVM 권한 설정

`qemu-system-x86_64` 실행 시 `permission denied /dev/kvm` 오류가 나면 현재 사용자를 kvm 그룹에 추가해야 한다.

<Terminal title="kvm 그룹 가입" commands={[
  { cmd: "sudo usermod -aG kvm $USER" },
  { cmd: "newgrp kvm" },
  { cmd: "ls -l /dev/kvm", output: ["crw-rw---- 1 root kvm  ...  /dev/kvm"] }
]} />

### 1-5. 이후 진행

이후 모든 단계는 WSL2 터미널에서 진행한다.
[Linux 셋업 가이드](/setup-linux)의 2단계(ISO 다운로드)부터 동일하게 따라가도 되고, 아래 요약 흐름을 그대로 써도 된다.

<Terminal title="작업 디렉터리 + ISO + 디스크" commands={[
  { cmd: "mkdir -p ~/qemu-lab && cd ~/qemu-lab" },
  { cmd: "wget -O ubuntu-22.04-server-amd64.iso https://releases.ubuntu.com/22.04/ubuntu-22.04.5-live-server-amd64.iso" },
  { cmd: "qemu-img create -f qcow2 ubuntu.qcow2 30G" },
  { cmd: "qemu-img create -f raw nvme_disk.raw 16G" }
]} />

<Callout kind="warn">
**WSL 환경에서 `-nographic` 주의** — 설치 단계에서 `-nographic`이 문제를 일으킬 수 있다.
이 경우 `-nographic`을 제거하면 WSLg가 QEMU GUI 창을 자동으로 띄운다.
설치 완료 후 GRUB 설정에 `console=ttyS0`을 추가하면 이후 `-nographic` 모드를 쓸 수 있다:

```bash
sudo sed -i 's/^GRUB_CMDLINE_LINUX=""/GRUB_CMDLINE_LINUX="console=tty0 console=ttyS0,115200n8"/' /etc/default/grub
sudo update-grub
sudo shutdown -h now
```
</Callout>

Ubuntu 설치:

<CodeBlock language="bash" filename="install-vm-wsl.sh">
{`qemu-system-x86_64 \\
  -m 4G -smp 4 \\
  -enable-kvm \\
  -drive file=ubuntu.qcow2,if=virtio,format=qcow2 \\
  -cdrom ubuntu-22.04-server-amd64.iso \\
  -boot d \\
  -netdev user,id=net0,hostfwd=tcp::2222-:22 -device virtio-net-pci,netdev=net0`}
</CodeBlock>

설치 완료 후 NVMe 포함 기동:

<CodeBlock language="bash" filename="run-vm-wsl.sh">
{`qemu-system-x86_64 \\
  -m 4G -smp 4 \\
  -enable-kvm \\
  -drive file=ubuntu.qcow2,if=virtio,format=qcow2 \\
  -drive file=nvme_disk.raw,id=nvm,if=none,format=raw \\
  -device nvme,id=nvme0,serial=deadbeef \\
  -device nvme-ns,drive=nvm,bus=nvme0,nsid=1,logical_block_size=512,physical_block_size=512 \\
  -netdev user,id=net0,hostfwd=tcp::2222-:22 -device virtio-net-pci,netdev=net0 \\
  -nographic`}
</CodeBlock>

<Terminal title="SSH 접속" commands={[
  { cmd: "ssh -p 2222 <username>@localhost" }
]} />

방법 1을 골랐다면 아래 "옵션 한 줄 해설" 섹션과 이후 게스트 안 단계로 바로 넘어간다.

## 방법 2: Windows 네이티브 QEMU + WHPX

WSL2를 사용하지 않고 Windows에서 직접 QEMU를 실행한다.
가속에는 **WHPX (Windows Hypervisor Platform)** 를 사용한다.

### 2-1. Windows 기능 활성화

PowerShell을 **관리자 권한**으로 실행:

<Terminal title="WHPX와 Hyper-V 활성화" commands={[
  { cmd: "dism.exe /online /enable-feature /featurename:HypervisorPlatform /all /norestart" },
  { cmd: "dism.exe /online /enable-feature /featurename:Microsoft-Hyper-V-All /all /norestart" }
]} />

재부팅 후 활성화 확인:

<Terminal title="WHPX 활성화 확인" commands={[
  { cmd: "Get-WindowsOptionalFeature -Online -FeatureName HypervisorPlatform | Select-Object State", output: ["State : Enabled"] }
]} />

### 2-2. QEMU 설치

#### 방법 A: Scoop (권장)

<Terminal title="Scoop 설치" commands={[
  { cmd: "Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser" },
  { cmd: "irm get.scoop.sh | iex" }
]} />

<Terminal title="QEMU 설치" commands={[
  { cmd: "scoop install qemu" }
]} />

#### 방법 B: Chocolatey

<Terminal title="Chocolatey 설치" commands={[
  { cmd: "Set-ExecutionPolicy Bypass -Scope Process -Force" },
  { cmd: "[System.Net.ServicePointManager]::SecurityProtocol = [System.Net.ServicePointManager]::SecurityProtocol -bor 3072" },
  { cmd: "iex ((New-Object System.Net.WebClient).DownloadString('https://community.chocolatey.org/install.ps1'))" }
]} />

<Terminal title="QEMU 설치" commands={[
  { cmd: "choco install qemu" }
]} />

#### 방법 C: 공식 인스톨러

[https://www.qemu.org/download/#windows](https://www.qemu.org/download/#windows) 에서 최신 Windows 인스톨러를 받아 설치.
설치 경로 예: `C:\Program Files\qemu`

PATH 추가 (PowerShell):

<Terminal title="PATH 추가" commands={[
  { cmd: "$env:PATH += \";C:\\Program Files\\qemu\"" },
  { cmd: "[System.Environment]::SetEnvironmentVariable(\"PATH\", $env:PATH, \"Machine\")" }
]} />

설치 확인:

<Terminal title="버전 확인" commands={[
  { cmd: "qemu-system-x86_64 --version" },
  { cmd: "qemu-img --version" }
]} />

### 2-3. Ubuntu 22.04 ISO 다운로드

<Terminal title="작업 폴더 + ISO 다운로드 (PowerShell)" commands={[
  { cmd: "mkdir $HOME\\qemu-lab" },
  { cmd: "cd $HOME\\qemu-lab" },
  { cmd: "curl.exe -L -o ubuntu-22.04-server-amd64.iso https://releases.ubuntu.com/22.04/ubuntu-22.04.5-live-server-amd64.iso" }
]} />

또는 브라우저에서 직접 받아 `C:\Users\<username>\qemu-lab\`에 둔다.

### 2-4. 디스크 이미지 생성

<Callout kind="note">
**디스크 분리 이유** — OS가 NVMe 위에 있으면 NVMe 드라이버 옵션을 바꿀 때 시스템이 멈출 수 있다.
OS는 virtio 디스크(`/dev/vda`)에, 실험 대상은 NVMe 디바이스(`/dev/nvme0n1`)에 분리해 둔다.
</Callout>

<Terminal title="디스크 이미지 생성" commands={[
  { cmd: "cd $HOME\\qemu-lab" },
  { cmd: "qemu-img create -f qcow2 ubuntu.qcow2 30G" },
  { cmd: "qemu-img create -f raw nvme_disk.raw 16G" }
]} />

### 2-5. Ubuntu 설치

<Callout kind="warn">
**Ubuntu 설치 시 주의사항**

- 설치 시에는 `-nographic`을 사용하지 않는다 (GUI 창으로 위저드 진행).
- `OpenSSH server` 반드시 선택.
- Storage: LVM 설정 시 디스크 전체를 사용하도록 설정. 기본값이 절반만 할당하므로, 설치 후 다음 명령으로 확장:

  ```bash
  sudo lvextend -l +100%FREE /dev/ubuntu-vg/ubuntu-lv
  sudo resize2fs /dev/ubuntu-vg/ubuntu-lv
  ```
- 설치 완료 후 QEMU 창을 닫거나 `Ctrl-A` + `X`로 종료.
- WHPX 가속 불가 시 오류가 나면 `-accel whpx` 대신 `-accel tcg`로 변경 (소프트웨어 에뮬레이션, 느림).
</Callout>

<CodeBlock language="powershell" filename="install-vm-windows.ps1">
{`cd $HOME\\qemu-lab

qemu-system-x86_64 \`
  -m 4G -smp 4 \`
  -accel whpx \`
  -drive file=ubuntu.qcow2,if=virtio,format=qcow2 \`
  -cdrom ubuntu-22.04-server-amd64.iso \`
  -boot d \`
  -netdev user,id=net0,hostfwd=tcp::2222-:22 -device virtio-net-pci,netdev=net0`}
</CodeBlock>

PowerShell 줄바꿈은 백틱(`` ` ``), CMD에서는 `^`다.

## 옵션 한 줄 해설

위 두 방법(WSL2/네이티브)에 등장한 QEMU 옵션을 정리한다.
WSL2와 네이티브의 차이는 가속 플래그 한 줄(`-enable-kvm` vs `-accel whpx`)뿐이다.

**리소스 할당 (공통)**

- `-m 4G` — 게스트에 할당할 RAM 크기.
- `-smp 4` — 게스트에 노출할 vCPU 수.

**가속 플래그**

- `-enable-kvm` (방법 1: WSL2) — Linux의 KVM. WSL2는 사실상 Linux이므로 KVM이 동작한다.
- `-accel whpx` (방법 2: 네이티브) — Windows Hypervisor Platform. Hyper-V 기능 활성화 필요.
- `-accel tcg` — 가속 없는 소프트웨어 에뮬레이션 (느림). 위 두 방법이 안 될 때 폴백.

**OS 디스크 (공통)**

- `-drive file=ubuntu.qcow2,if=virtio,format=qcow2` — Ubuntu가 설치되는 OS 디스크.
  - `if=virtio` — virtio 인터페이스 (`/dev/vda`로 노출). NVMe와 분리.
  - `format=qcow2` — Copy-on-Write 포맷.

**ISO 관련 (설치 단계만)**

- `-cdrom <iso>` — ISO를 가상 CD-ROM으로 마운트.
- `-boot d` — CD-ROM에서 부팅하라는 명시적 지정. 빠지면 빈 OS 디스크에서 부팅하려다 UEFI 셸로 떨어진다.

**NVMe 디스크 (기동 단계만)**

NVMe는 세 줄로 분리해서 정의한다.

- `-drive file=nvme_disk.raw,id=nvm,if=none,format=raw` — 백킹 스토리지 등록.
  - `if=none` — 어떤 인터페이스에도 자동 연결 안 함. 다음 줄의 `-device`가 가져다 쓴다.
  - `format=raw` — `O_DIRECT` 정확히 동작하려면 raw 필수.
- `-device nvme,id=nvme0,serial=deadbeef` — NVMe 컨트롤러.
- `-device nvme-ns,drive=nvm,bus=nvme0,nsid=1,logical_block_size=512,physical_block_size=512` — namespace 추가, 위 `nvm` 드라이브 연결.

세 줄이 모두 있어야 게스트에 `/dev/nvme0n1`이 보인다.

**네트워크 (공통)**

- `-netdev user,id=net0,hostfwd=tcp::2222-:22` — user-mode 네트워크 백엔드(NAT). 게스트가 호스트 인터넷을 빌려 쓴다.
- `-device virtio-net-pci,netdev=net0` — virtio-net 가상 NIC를 위 백엔드에 연결. Ubuntu 커널에 virtio 드라이버가 기본 포함이라 즉시 인식한다.
- `hostfwd=tcp::2222-:22` — 호스트의 2222 포트를 게스트의 22(SSH)로 연결.

레거시 `-net nic -net user,...`로도 동작하지만, 게스트가 e1000으로 부팅돼 `Wait for Network to be Configured`에서 30~120초 멈추는 사례가 많다. 위 `-netdev + virtio-net-pci` 형태가 안정적이다.

**디스플레이 (공통)**

- `-nographic` — GUI 창 없이 시리얼 콘솔을 현재 터미널에 직결. 종료는 `Ctrl-A` + `X`.
- 방법 1(WSL2)에서 설치 단계엔 `-nographic`을 빼야 WSLg GUI 창이 뜬다.
- 방법 2(네이티브)에서 설치 단계엔 `-nographic`을 빼야 QEMU GUI 창이 뜬다.

## 트러블슈팅

| 증상 | 원인 | 해결 |
|---|---|---|
| `WHPX: Failed to initialize` | Windows Hypervisor Platform 미활성 | PowerShell 관리자로 `HypervisorPlatform` 기능 활성화 후 재부팅 |
| `WHPX not supported` | Hyper-V와 충돌 또는 하드웨어 미지원 | `-accel tcg`로 변경 (느림) |
| `qemu-system-x86_64: command not found` | PATH 미등록 | QEMU 설치 경로를 PATH에 추가 |
| PowerShell 줄바꿈 오류 | 백틱(`` ` ``) 뒤 공백 | 백틱 뒤에 공백·탭 없도록 확인 |
| `SSH Connection refused` | VM 부팅 중 | 30~60초 대기 후 재시도 |
| `nvme0n1` 없음 | `-device nvme` 누락 | QEMU 명령의 NVMe 세 줄 확인 |
| WSL2에서 `/dev/kvm` 없음 | Hyper-V 비활성 | `HypervisorPlatform` 및 `Microsoft-Hyper-V` 기능 활성화 |
| PuTTY `Connection timed out` | 포트 포워딩 오류 | QEMU에서 `hostfwd=tcp::2222-:22` 확인, Windows 방화벽 확인 |
| ISO 부팅 실패 | `-boot d` 누락 | 설치 명령에 `-boot d` 포함 확인 |
| SSH `WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED` | 이전 VM의 호스트 키와 충돌 | `ssh-keygen -R "[localhost]:2222"` |
| 부팅 시 `Wait for Network to be Configured` 30~120초 정지 / 게스트에서 `apt install`이 DNS 오류로 실패 | 레거시 `-net nic`로 e1000 NIC을 받았는데 netplan이 인터페이스를 못 잡음 | (1) QEMU 명령을 위 `-netdev user,id=net0,... -device virtio-net-pci,netdev=net0` 형태로 교체. (2) 이미 설치된 VM이라면 게스트에서 `ip a`로 인터페이스 이름 확인 → `/etc/netplan/00-installer-config.yaml`에서 인터페이스 이름을 실제 이름으로 수정 → `sudo netplan apply`. (3) 부팅 대기를 단축하려면 `sudo systemctl disable systemd-networkd-wait-online.service` |

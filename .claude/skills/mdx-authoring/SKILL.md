# MDX Authoring

## 이 프로젝트의 MDX는 무엇인가

이 페이지는 8개 MDX 파일이 순서대로 조립되어 하나의 긴 글을 이룬다. 각 MDX는 독립된 글이 아니라 **장(chapter)**이며, 앞 장의 결론을 받아 다음 장의 질문으로 넘어간다.

## 본문 톤

- 한국어 줄글
- 이모지 없음
- "우리는", "살펴보자" 같은 강의체보다 "이 절에서는 ~를 다룬다", "~의 동작을 정리하면 다음과 같다" 같은 서술체를 선호
- 한 문단은 3~6문장. 한 문장 안에 두 가지 주장을 넣지 않는다
- 불릿은 정말 병렬 항목일 때만. "장점 3가지" 같은 자연스러운 병렬은 OK, 줄글로 풀 수 있는 건 줄글로

## 회차 섹션의 고정 구조

`03-week1.mdx`, `04-week2.mdx`, `05-week3.mdx`는 항상 다음 다섯 블록을 가진다. 순서를 바꾸지 않는다.

1. **목표** — 이 회차에서 무엇을 보이려 하는가 (1문단)
2. **개념** — 필요한 배경지식 (2~4문단)
3. **코드** — 실제 구현. `<CodeBlock language="c">`로 감싸고 빌드 명령을 함께 둔다
4. **실험 방법** — 커널 버전, 디스크, fio/stress-ng 명령까지 재현 가능 수준으로
5. **결과** — `<Chart>` 또는 표 + 해석 문단

## MDX 안의 컴포넌트

다음 컴포넌트만 사용한다. 새 컴포넌트가 필요하면 먼저 `src/components/`에 만들고 import 한다.

- `<Section title="..." id="...">` — 섹션 래퍼. 모든 본문은 이 안에 들어간다
- `<CodeBlock language="c" filename="bench.c">...</CodeBlock>`
- `<Chart kind="line" dataKey="latency" data={qdLatency} />`
- `<Callout kind="note|warn">...</Callout>` — 보충 설명. 회차당 1개 이하

데이터는 `src/data/experiments.ts`에서 import 한다. MDX 안에 raw 데이터 배열을 박지 않는다.

## 헤딩 레벨

각 MDX는 `##`부터 시작한다. `#`은 페이지 전체 제목이라 `index.tsx`의 Hero가 가진다. 회차 안의 "목표/개념/코드/실험 방법/결과"는 `###`을 쓴다.

## 코드 블록 규칙

- C 코드는 항상 빌드 명령을 본문이나 코드 블록 직전에 명시
- 코드 안 주석은 영어
- `// TODO`, `// FIXME` 같은 흔적을 남기지 않는다
- 80자 안쪽으로 wrap

## 자가 점검

- [ ] 이전 장의 결론에서 자연스럽게 이어지는 첫 문단인가
- [ ] 회차 섹션이라면 다섯 블록이 모두 있는가
- [ ] 이모지가 없는가
- [ ] 불릿보다 줄글이 많은가
- [ ] 학부 3학년이 첫 문단을 읽고 막히지 않는가
- [ ] 수치가 NVMe/SATA SSD의 현실 범위를 벗어나지 않는가
